# 任务管理设计文档

> 文档版本：v1.0  
> 最后更新：2026-04-08

## 设计原则

总体设计原则：做一个**轻量的任务管理**，满足基本使用要求即可。

任务在保险代理人助手的产品定位中，是"顾问式雅致工作台"中的**经营节奏管理工具**，而非复杂的项目管理或协同系统。任务的核心价值是帮助代理人跟踪客户经营中的后续待办事项，确保跟进节奏不被遗漏。

---

## 一、数据结构

### 1.1 任务字段定义

任务的内容包含以下字段：

| 字段名 | 类型 | 说明 | 必填 |
|--------|------|------|------|
| `id` | uuid | 任务编号，主键 | 系统自动生成 |
| `owner_id` | uuid | 任务创建人（归属人） | 系统自动填写 |
| `customer_id` | uuid | 关联客户 ID | 可空 |
| `title` | text | 任务标题 | **必填** |
| `note` | text | 任务备注/描述 | 可空 |
| `planned_at` | timestamptz | 计划执行时间 | **必填** |
| `remind_at` | timestamptz | 提醒时间 | 可空，默认无值 |
| `completed_at` | timestamptz | 完成时间 | 标记完成时自动填写 |
| `canceled_at` | timestamptz | 取消时间 | 标记取消时自动填写 |
| `priority` | text | 优先级：`高` / `中` / `低` | 默认`中` |
| `status` | text | 状态：`待开始` / `已完成` / `已取消` | **必填** |
| `source_type` | text | 来源类型：`manual` / `visit` / `activity` | 必填 |
| `source_id` | uuid | 来源记录 ID | 可空 |
| `created_at` | timestamptz | 创建时间 | 系统自动填写 |

**注**：所有涉及时间的字段数据类型均为 `timestamptz`，服务端统一保存标准时间戳。

### 1.2 来源类型定义

| source_type | 含义 | source_id |
|-------------|------|-----------|
| `manual` | 手工创建任务 | `null` |
| `visit` | 来源于拜访记录 | 对应拜访记录 ID |
| `activity` | 来源于活动记录 | 对应活动记录 ID |

### 1.3 必填项规则

1. **任务标题**、**任务的计划执行时间** 为必填项
2. **任务编号**、**任务的创建时间**、**任务的创建人**、**任务的状态**，在任务创建时由系统自动填写

### 1.4 唯一性 / 去重规则

任务保存时的唯一性判断以本规则为准：

**（1）用户手工创建的任务**
```
owner_id + title + planned_at + coalesce(customer_id, '00000000-0000-0000-0000-000000000000')
```

**（2）来源于拜访/活动的任务**
```
owner_id + source_type + coalesce(source_id, '00000000-0000-0000-0000-000000000000') + title + coalesce(customer_id, '00000000-0000-0000-0000-000000000000')
```

**业务含义说明**：对来源任务而言，去重优先保证"同一来源事项不被重复创建"；因此同一来源、同一客户、同一标题的任务视为同一任务，不再以 `planned_at` 区分多条。

### 1.5 时间处理规则

- 所有时间字段以 `timestamptz` 由服务端统一落库存储标准时间戳
- 页面展示，以及"今天 / 当前时间 / 是否过期 / 今日提醒"的判断，**按用户当前业务时区换算和计算**

## 二、任务创建

### 2.1 创建入口

有 3 个入口可以创建任务：

| 入口 | 流程 | 客户关联 |
|------|------|----------|
| **拜访记录** | 保存拜访记录后，若识别到后续工作，自动生成待确认任务草稿，用户修改确认后正式创建 | 自动与拜访记录匹配的客户关联 |
| **活动记录** | 保存活动记录后，若识别到后续工作，自动生成待确认任务草稿，用户修改确认后正式创建 | 自动与活动记录匹配的客户关联 |
| **手工新增** | 单独的添加任务入口，用户自行创建任务 | 用户可自主选择是否匹配现有客户 |

### 2.2 任务草稿与确认规则

**核心原则**：来源于拜访/活动的待确认任务草稿，必须经用户修改并确认后再正式创建。

- **来源任务草稿**：保存拜访/活动记录后，助手自动识别后续事项 → 生成待确认任务草稿 → 用户修改确认 → 正式创建任务
- **中断可恢复**：来源于拜访/活动的待确认任务草稿，如用户暂未确认创建，系统需保留草稿并支持后续恢复继续确认

### 2.3 提醒时间

- 任务的提醒时间，**默认无值**，由用户自行填写

### 2.4 优先级

- 优先级包括 `高`、`中`、`低` 三级
- 创建时默认 `中` 优先级，用户可自行修改

### 2.5 任务状态

任务的状态包括 **3 种**：

| 状态 | 触发时机 |
|------|----------|
| `待开始` | 创建任务时自动生成该状态 |
| `已完成` | 由用户自行修改任务状态为已完成 |
| `已取消` | 由用户自行修改任务状态为已取消 |

**注**：`已过期` 不是数据库存储状态，而是页面计算展示分区。

### 2.6 计划执行时间限制

- **允许创建计划执行时间早于当前时间的任务**
- 创建该类任务后，在**已过期任务区**中展示
- "当前时间"取用户当前业务时区的时间

## 三、任务查看

查看任务时，分为**两个大区**：

### 3.1 顶部：今日提醒聚焦区

**展示条件**：
- 当前日期与任务提醒时间的日期相同
- 任务状态是 `待开始`

**特殊规则**：
- 顶部提醒聚焦区**允许**与主分区重复展示
- 聚焦区是提醒视图，不改变任务在主分区中的归属
- "当前日期"取用户当前业务时区的日期

### 3.2 主分区

#### （1）待开始任务区

**展示条件**：
- 任务的计划执行时间**大于**当前时间
- 任务状态是 `待开始`

- "当前时间"取用户当前业务时区的时间

#### （2）已过期任务区

**展示条件**：
- 任务的计划执行时间**小于等于**当前时间
- 任务状态是 `待开始`

- "当前时间"取用户当前业务时区的时间

#### （3）已完成任务区

**展示条件**：
- 任务状态为 `已完成`

#### （4）已取消任务区

**展示条件**：
- 任务状态为 `已取消`

### 3.3 时间判断规则总结

| 场景 | 判断依据 |
|------|----------|
| 今日提醒聚焦区 | `remind_at` 的日期部分 = 当前日期（用户时区） |
| 待开始任务区 | `planned_at > 当前时间`（用户时区） |
| 已过期任务区 | `planned_at <= 当前时间`（用户时区） |

## 四、任务修改

### 4.1 可修改字段

用户在查看任务时，可以对任务进行以下修改：

| 修改项 | 说明 |
|--------|------|
| 修改任务的提醒时间 | 可修改或清空 |
| 修改任务的优先级 | 可在 `高/中/低` 之间切换 |
| 修改任务的计划执行时间 | 可修改为过去或未来时间 |
| 修改任务标题 | 可修改标题内容 |
| 修改任务备注 | 可修改备注内容 |

### 4.2 状态变更操作

#### （1）标记为已完成

- 用户可将任务状态修改为 `已完成`
- 修改后，任务状态改为 `已完成`
- 系统自动记录任务的完成时间（取用户当前业务时区的时间）

#### （2）标记为已取消

- 用户可将任务状态修改为 `已取消`
- 修改后，任务状态改为 `已取消`
- 系统自动记录任务的取消时间（取用户当前业务时区的时间）

### 4.3 修改范围限制（第一阶段）

- **第一阶段，任务修改动作默认只针对 `待开始` 任务**
- `已完成`、`已取消` 的任务视为结束任务，默认只读
- 第一阶段不支持将结束任务恢复为 `待开始`

---

## 五、来源记录变更后的任务处理规则

### 5.1 首次保存拜访/活动

```
提取后续事项 → 生成待确认任务草稿 → 用户确认后创建任务
```

### 5.2 后续再次编辑拜访/活动

- **不静默覆盖已存在任务**
- 如识别出新的后续事项，生成"新增待确认任务草稿"
- 如原后续事项被删掉，**也不自动删除**已存在任务
- 由用户自行决定是否取消已有任务

### 5.3 删除拜访/活动

- **不自动删除任务**
- 任务一旦被确认创建，即视为独立经营事项，与来源记录解耦

---

## 六、任务结束

**定义**：当任务的状态为 `已完成` 或 `已取消` 时，视为任务结束。

**结束任务的表现**：
- 结束任务默认只读（第一阶段）
- 结束任务不再参与"今天/当前时间/是否过期/今日提醒"的计算
- 结束任务保留历史记录，便于后续复盘

## 附录：与现有设计的核心差异

| 维度 | 现有设计 | 本设计 |
|------|----------|--------|
| 核心时间字段 | `due_date` / `due_at` / `remind_at` | `planned_at` / `remind_at` / `completed_at` / `canceled_at` |
| 存储状态 | 5 个（含 `已逾期`） | 3 个（`待开始/已完成/已取消`） |
| `已过期` | 数据库状态 | 页面计算分区 |
| 来源任务生成 | 直接同步入库 | 生成草稿 → 用户确认 → 正式创建 |
| 结束任务修改 | 未限制 | 第一阶段只读 |
| 来源变更处理 | 未明确 | 不自动覆盖/删除 |
